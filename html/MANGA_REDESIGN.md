# 漫画版排版改造规划

把 manga 分支的前端从「通用文件浏览器」改造成「真正的漫画网站」。

> **范围约束（已定）**：纯前端渲染升级。后端 `/list`、`/files/`、`/thumb/` 一律不动，
> 守住 bakaWFS「文件系统即唯一状态来源、无数据库」哲学。所有漫画语义在前端从
> Node 树 + article.md 推导，不在后端建索引/元数据 API。
> **阅读模式（已定）**：翻页（日漫/同人志），复用现有 viewer。

## 数据基础（已勘察）

```
/opt/漫画/
├── <汉化组>/                    第1层：children 含 type:dir → 分组节点
│   ├── <作品>/                  第2层：children 全是 type:file 图片(+article.md) → 漫画本
│   │   ├── 1.jpg ~ N.jpg        页面图
│   │   └── article.md          元数据（标题/来源/汉化组/gid/页数）
│   └── <连载作品>/              变体：children 含 type:dir「第N话」→ 递归分组
```

`/list` 返回的 Node：`{name, type:"file"|"dir", size, children}`。前端据此判定节点性质，
**无需后端任何改动**。

article.md 元数据格式（前端可解析）：
```
# <标题>
> 来源：e-hentai | <url> | 抓取于 <date>
> 汉化组：<组名>
> gid：<id>
> 页数：<n>
```

## 节点判定规则（前端）

| 节点 | 判定 | 渲染为 |
|------|------|--------|
| 漫画本 | children 全是图片 file（忽略 article.md/.json/.log） | 封面卡片，点击→阅读器 |
| 分组目录 | children 含 type:dir | 可进入的分组（汉化组 / 连载作品） |
| 普通文件 | 非图片、非 article.md | 保持原文件卡片（向后兼容） |

封面 = 该漫画本目录里排序第一张图（通常 `1.jpg`），复用现有 `/thumb/` 缩略图缓存。

## 三层视觉改造

### 第1层 汉化组 → 出版方卡片
- 现状：5 个 📁 文件夹图标
- 改成：每组一张卡片 = 组名 + 作品数 + 封面墙（组内前 3~4 本的封面缩略拼图）

### 第2层 作品列表 → 漫画封面网格（核心视觉）
- 现状：文件夹文字列表
- 改成：响应式封面网格。每本 = 封面图(1.jpg via /thumb) + 标题 + 页数徽章 + 来源徽章(e-h/B站)
- 标题/页数/来源优先取自 article.md，缺失则回退目录名 + 实际图片数

### 第3层 进入作品 → 直接阅读器
- 现状：点进去看到 `1.jpg 2.jpg...` 文件列表，需再点图
- 改成：点封面**直接全屏翻页阅读**（复用现有 viewer，少一层中转）
- article.md 不作为文件展示（识别并隐藏，其内容用作元数据）

## 复用现有能力（不重造）

| 需求 | 复用 |
|------|------|
| 封面缩略图 | 现有 `/thumb/` + `.thumbcache`（内容 hash 缓存，已验证可跨机复用） |
| 翻页阅读 | 现有 viewer（image 模式，预加载 + 预解码 + key 重建，已很完善） |
| 列表缩略图批量 | 现有 `/thumbs` 批量接口 + parseThumbStream |
| 图片识别 | utils.js `classifyFile` / `isImageFile` |
| 排序 | utils.js `naturalCompare`（四段分组，刚改） |

## 实现阶段（建议）

| 阶段 | 内容 | 风险 |
|------|------|------|
| 1 | article.md 解析器（前端 JS，提取标题/页数/来源）+ 节点判定函数 | 低 |
| 2 | 第2层封面网格：漫画本渲染成封面卡片（取 1.jpg），替换文件夹列表 | 中 |
| 3 | 第3层：点漫画本封面直接进 viewer，article.md 隐藏 | 低 |
| 4 | 第1层：汉化组卡片 + 封面墙拼图 | 中 |
| 5 | 富化：页数/来源徽章、连载作品「第N话」分组优化 | 低 |

每阶段独立可用、可回退。阶段 1~3 完成即「能像漫画站一样浏览+阅读」，4~5 是锦上添花。

## 不做（守住边界）

- ❌ 后端解析 article.md / 加元数据 API（破坏无状态哲学）
- ❌ 后端建作品索引 / 数据库
- ❌ 条漫（竖刷）模式 —— 本批只做翻页
- ❌ 收藏/历史/进度记忆 —— 需引入"另一个状态来源"，本批不碰
